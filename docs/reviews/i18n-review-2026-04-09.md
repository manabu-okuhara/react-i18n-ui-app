# I18n Review

Date: 2026-04-09
Reviewer: Codex
Scope: Review of the current React app with emphasis on locale resources, hard-coded UI text, formatting behavior, RTL handling, and test coverage.

## Summary

- Overall status: The project has a working custom i18n structure with locale JSON files, URL-driven locale selection, locale-aware formatting, and basic RTL support.
- Main risks: Several visible strings still bypass the locale files, one locale file is largely untranslated, and one plural message is malformed and likely to fail for Russian.
- Recommended next step: Fix the broken and untranslated locale resources first, then remove hard-coded UI strings from components and update tests to assert localized behavior.

## Findings

### 1. Russian locale is mostly still English
- Severity: High
- File: `src/locales/ru-RU.json`
- Lines: 2-7
- Issue: The Russian locale file keeps the title, locale prompt, date sentence, country sentence, number sentence, and share label in English. Selecting `ru-RU` therefore produces a partially untranslated UI.
- Recommendation: Translate every user-facing key in `src/locales/ru-RU.json` so the locale is complete and consistent.

### 2. Russian plural message is malformed
- Severity: High
- File: `src/locales/ru-RU.json`
- Lines: 8
- Issue: The plural string repeats `count, plural` inside the message, which does not match the message structure expected by `formatMessage` in `src/Dashboard.js`. This can break plural rendering or produce incorrect Russian output.
- Recommendation: Rewrite the Russian plural message into a single valid plural pattern with `one`, `few`, `many`, and `other` branches.

### 3. App shell still contains hard-coded English starter text outside the locale system
- Severity: Medium
- File: `src/App.js`
- Lines: 12-22
- Issue: The `alt` text for the logo, the "Edit src/App.js and save to reload." copy, and the "Learn React" link text are rendered directly in JSX instead of coming from locale resources.
- Recommendation: Either remove the Create React App starter content or move these strings into the existing locale files and message flow.

### 4. Dashboard includes hard-coded English fallback strings and interaction text
- Severity: Medium
- File: `src/Dashboard.js`
- Lines: 27-34, 127-129, 141, 149
- Issue: The initial label state includes English copy, the copy confirmation uses `alert('Link copied!')`, the loading title is `'...'`, and the search input placeholder is hard-coded as `e.g. de-DE, ja-JP`. These strings bypass translation resources and will remain English regardless of locale.
- Recommendation: Move fallback and interaction strings into locale resources or derive them from the active messages object.

### 5. French locale contains a visible typo in the title
- Severity: Low
- File: `src/locales/fr-CA.json`
- Lines: 2
- Issue: The title begins with `TTableau` instead of `Tableau`, so users will see a typo in the French UI.
- Recommendation: Correct the translated string in the locale file.

### 6. Tests do not cover the current i18n behavior
- Severity: Low
- File: `src/App.test.js`
- Lines: 4-7
- Issue: The only test still checks for the legacy "Learn React" link. It does not verify locale selection, locale file rendering, fallback to `en-US`, pluralization, or RTL behavior.
- Recommendation: Replace the starter test with focused tests for localized rendering and locale switching behavior.

## Positive Notes

- Locale resources are already separated into `src/locales/*.json`, which gives the project a clear extension point.
- `src/Dashboard.js` uses browser `Intl` APIs for number, plural, region, and date formatting instead of manual string assembly.
- The selected locale is persisted in the `hl` query parameter, which makes locale state shareable.
- The dashboard applies `dir` and alignment changes for RTL locales, which is a strong foundation for Arabic support.

## Testing Gaps

- Locale switching is not covered by automated tests.
- Fallback locale behavior to `en-US` is not covered by automated tests.
- Pluralization behavior across multiple locales is not covered by automated tests.
- RTL layout behavior is not covered by automated tests.
- Formatting for dates and numbers is not covered by automated tests.
